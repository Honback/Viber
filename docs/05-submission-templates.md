# 등록 템플릿 명세

## 1. 등록 설계 원칙

- 입력은 많아 보이지 않게 하되 결과물은 풍부하게 보여야 한다.
- 모든 유형은 프로젝트 중심 구조를 유지해야 한다.
- Launch, Update, Feedback 세 가지 유형만 허용한다.
- 필수 항목은 품질 유지에 필요한 최소 수준으로 제한한다.
- 제출 마지막 단계에서 실제 미리보기를 보여준다.

## 2. 공통 입력 원칙

### 공통 필수

- 프로젝트 또는 활동 유형
- 프로젝트명 또는 기존 프로젝트 식별
- 대표 링크
- 최소 1개 이상의 설명성 텍스트
- 메이커 식별 수단

### 공통 권장

- 대표 이미지
- 실제 체험 가능한 링크
- 왜 만들었는지
- 사용 대상 설명

### 공통 자동 수집

가능하면 아래 메타데이터를 링크로부터 자동 수집한다.

- `og:title`
- `og:description`
- `og:image`

자동 수집값은 사용자가 수정 가능해야 한다.

## 3. Launch 템플릿

### 목적

새 프로젝트를 처음 공개하는 제출 유형이다.

### 필수 항목

- 프로젝트명
- 한 줄 소개
  - 권장 최대 60자
- 짧은 설명
  - 권장 최대 200자
- 라이브 URL
- 대표 이미지
- 어떤 문제를 푸는지
- 누구를 위한 것인지
- 현재 상태
  - alpha
  - beta
  - live
- 카테고리
- 플랫폼
- maker alias

### 선택 항목

- GitHub URL
- 데모 영상
- 가격 정보
- 사용한 AI 툴
- 오픈소스 여부
- 가입 없이 체험 가능 여부
- 왜 만들었는지
- 추가 이미지
- 문서 링크

### 제출 품질 기준

- 랜딩 페이지만 있고 제품 체험이 불가능하면 보류 대상이다.
- 프로젝트 소개만 있고 실제 링크가 죽어 있으면 등록 불가다.
- 대표 이미지는 프로젝트 카드 품질에 직접 영향을 준다.

## 4. Update 템플릿

### 목적

기존 프로젝트의 변경 사항을 기록한다.

### 필수 항목

- 무엇을 바꿨는지
- 왜 바꿨는지
- 스크린샷 또는 GIF 1개 이상

### 선택 항목

- 이전 버전 대비 핵심 차이
- 사용자 피드백 반영 여부
- 관련 링크
- 성능 또는 지표 변화

### 권장 길이

- 제목: 한눈에 바뀐 점이 보일 정도로 짧게
- 요약: 2~3문장
- 본문: 상세 변경점 정리

## 5. Feedback 템플릿

### 목적

- owner는 사용자에게 특정 검증 포인트를 요청하는 `Ask for Feedback`으로 사용한다.
- member는 구조화된 사용 후기나 제품 피드백을 남기는 `Feedback`으로 사용한다.

### owner 작성 시 필수 항목

- 지금 어떤 피드백이 필요한지
- 어떤 사용자에게 보고 싶은지
- 무엇을 눌러봐 달라는지
- 최소 1개 이상의 미디어 또는 프로토타입 링크

### owner 작성 시 선택 항목

- 2~3개의 구체 질문
- 현재 막히는 지점
- 비교 중인 옵션

### member 작성 시 필수 항목

- 무엇을 직접 써봤는지
- 어느 지점이 좋았는지 또는 막혔는지
- 가능하면 재현 맥락 또는 스크린샷 1개 이상

### member 작성 시 선택 항목

- 개선 제안
- 비교한 다른 제품
- 반복 사용 의사 또는 추천 여부

### 좋은 피드백 요청의 조건

- 요청 대상이 구체적이어야 한다.
- "한번 써주세요"보다 "온보딩 1단계가 헷갈리는지 봐주세요"처럼 구체적이어야 한다.
- 실제로 눌러볼 수 있어야 한다.
- member 피드백은 감상 한 줄보다 재현 가능한 맥락이 있어야 한다.

## 6. 메이커 정보 입력

### 필수

- maker alias
- 이메일 매직링크 또는 GitHub 인증 중 하나

### 선택

- GitHub 프로필 링크
- X 링크
- 개인 웹사이트

### 원칙

- 공개 이름은 유연하게 허용한다.
- 내부 소유권은 반드시 확보한다.

## 7. 미디어 입력 규칙

### 기본 규칙

- 대표 이미지 1개 필수
- 추가 이미지 2~5개 권장
- GIF 또는 짧은 영상은 선택

### 목적

- 카드와 상세 상단에서 즉시 이해 가능한 화면을 제공한다.
- 텍스트보다 먼저 프로젝트를 설명하게 만든다.

## 8. 검증과 보류 규칙

제출 시점에 아래 항목은 자동 검사 또는 운영 후속 점검 대상이다.

- 링크 접근 가능 여부
- 동일 URL 중복 여부
- 금칙어
- 지나치게 빈약한 설명
- 과도한 외부 링크
- 사칭 의심

## 9. 미리보기 규칙

발행 전에는 최소 아래 미리보기를 보여준다.

- 카드 미리보기
- 프로젝트 상세 상단 미리보기
- 태그와 배지 노출 상태

사용자가 어떤 모습으로 공개되는지 이해한 상태에서 제출하도록 만든다.

## 10. 운영 관점 메모

- 입력 항목을 너무 많이 강제하면 등록 전환율이 떨어진다.
- 입력 항목을 너무 느슨하게 두면 탐색 품질이 망가진다.
- 따라서 "필수는 적게, 구조는 강하게"가 원칙이다.
