# 페이지 명세

## 1. 홈 `/`

### 목적

- 좋은 프로젝트를 빠르게 발견하게 만든다.
- 사용자가 읽기 전에 먼저 써보게 만든다.
- 서비스의 정체성을 "게시판"이 아니라 "프로젝트 쇼케이스"로 전달한다.

### 레이아웃

1. 상단 헤더
   - 로고
   - 검색
   - `탐색`
   - `등록하기`
   - 로그인 또는 `내 저장`

2. 히어로 영역
   - 한 줄 메시지
   - 보조 설명
   - CTA 두 개
     - `프로젝트 둘러보기`
     - `내 프로젝트 올리기`

3. 추천 프로젝트
   - 에디터 추천 3~6개
   - 큰 카드 또는 혼합 카드 레이아웃

4. 새로 나온 프로젝트
   - 최신 `Launch` 중심 카드 그리드

5. 피드백 기다리는 프로젝트
   - `Ask for Feedback` 활동이 최근인 프로젝트

6. 이번 주 업데이트
   - 최근 `Update`가 붙은 프로젝트

7. 빠른 필터 칩
   - Web
   - Mobile
   - Open Source
   - No Signup
   - Free
   - Beta

8. 푸터

### UX 원칙

- 첫 화면 안에서 실제 프로젝트 카드가 보여야 한다.
- 카드 안 `Try` 버튼이 명확해야 한다.
- 히어로보다 카드의 체험 유도가 더 중요하다.
- 최신 글 목록처럼 텍스트가 길게 이어지면 안 된다.

## 2. 탐색 페이지 `/projects`

### 목적

- 전체 프로젝트를 검색, 정렬, 필터링하는 메인 탐색 화면이다.

### 필수 요소

- 검색 입력
- 정렬 드롭다운
- 필터 칩 또는 필터 패널
- 결과 수 표시
- 카드 그리드 또는 리스트
- 페이지네이션 또는 무한 스크롤

### 정렬 기준

- 최신순
- 트렌딩
- 최근 업데이트
- 댓글 많은 순

### 필터

- 카테고리
- 플랫폼
- 스테이지
- 가격 모델
- Open Source
- No Signup
- Solo Maker
- 활동 유형

### 상태 처리

- 결과 없음
- 검색어 없음
- 로딩 스켈레톤
- 오류 상태

### 모바일

- 필터는 하단 시트 또는 전체 화면 오버레이로 연다.
- 정렬은 상단 sticky 바에 유지한다.

## 3. 프로젝트 상세 `/p/[slug]`

### 목적

- 프로젝트 이해, 체험, 피드백, 히스토리 추적을 한 페이지 안에서 해결한다.

### 첫 화면 안 필수 요소

- 프로젝트명
- 한 줄 소개
- 상태 배지
- `Try`
- `GitHub`
- `Save`
- `Share`
- `Report`
- maker alias
- 검증 상태
- 마지막 업데이트일

### 본문 구성

1. 대표 미디어
   - 썸네일
   - 스크린샷
   - GIF
   - 짧은 데모

2. Overview
   - 무엇인지
   - 어떤 문제를 푸는지
   - 누구를 위한 것인지

3. Why I made this
   - 왜 만들었는지
   - 바이브코딩 맥락

4. Current Status
   - alpha/beta/live
   - 가격
   - 가입 필요 여부
   - 오픈소스 여부

5. Activity Feed
   - Launch
   - Update
   - Ask for Feedback / Feedback

6. Comments / Community Feedback
   - 프로젝트 전체 댓글
   - 활동 단위 맥락 포함 가능
   - visitor 댓글은 guest alias + CAPTCHA 기반
   - 인증 member는 구조화된 feedback 활동 작성 가능

7. Related Projects
   - 같은 태그 또는 카테고리 기반

### 메타 정보

- 카테고리
- 플랫폼
- 가격 모델
- 사용한 AI 툴
- 링크 모음
- 검증 상태

### UX 원칙

- `Try`는 스크롤 전에도 보여야 한다.
- 상세 페이지가 블로그 포스트처럼 길어지면 안 된다.
- 변경 이력이 분명히 보여야 신뢰가 생긴다.

## 4. 등록 페이지 `/submit`

### 목적

- 긴 폼이 아니라 단계형 제출 경험을 제공한다.

### 단계 구성

#### Step 1. 유형 선택

- 새 프로젝트 런치

기존 프로젝트의 `Update`는 `/submit`에서 받지 않는다.
owner의 `Ask for Feedback`는 `/me/projects`에서 시작하고, member의 `Feedback`은 프로젝트 상세에서 시작한다.

#### Step 2. 기본 링크 입력

- Live URL
- GitHub URL
- Demo 영상 링크 optional

가능하면 여기서 OG 메타데이터를 자동 수집한다.

#### Step 3. 프로젝트 기본 정보

- 프로젝트 이름
- 한 줄 소개
- 짧은 설명
- 어떤 문제를 푸는지
- 누구를 위한 것인지
- 현재 상태
- 가격 모델
- 카테고리
- 플랫폼

#### Step 4. 미디어

- 대표 이미지 1개 필수
- 추가 이미지 2~5개
- GIF 또는 짧은 영상 optional

#### Step 5. 메이커 정보

- maker alias
- GitHub 연결 또는 이메일 매직링크
- 선택적 소셜 링크

#### Step 6. 미리보기 및 발행

- 최종 미리보기
- 운영 정책 체크
- Publish

### UX 원칙

- 입력 이유가 분명해야 한다.
- 필수 항목은 최소화하되 품질은 유지해야 한다.
- 마지막에 실제 카드/상세 미리보기가 보여야 한다.

## 5. Claim 페이지 `/claim/[token]`

### 목적

- 비회원 제출 프로젝트의 수정권을 안전하게 연결한다.

### 기능

- claim token 검증
- 소유권 연결 완료 안내
- 기존 프로젝트 편집으로 이동
- 만료 또는 잘못된 토큰 예외 처리

## 6. 내 저장 `/me/saved`

### 목적

- 저장한 프로젝트를 다시 쉽게 방문하게 만든다.

### 기능

- 저장한 프로젝트 목록
- 최근 저장 순 정렬
- 태그와 상태 표시
- 저장 취소

## 7. 내 프로젝트 `/me/projects`

### 목적

- 내가 관리하는 프로젝트를 한곳에서 관리한다.

### 기능

- 프로젝트 목록
- 상태 표시
  - pending
  - published
  - limited
  - archived
- 프로젝트 수정
- 새 활동 추가
- 공개 상태 확인

## 8. 관리자 운영 `/admin/moderation`

### 목적

- 공개 이후 발생한 신고, 스팸, 중복, 죽은 링크, 운영 이슈를 검토하고 조치한다.

### 기능

- 신고 큐
- 운영 이슈 큐
- 필터
  - 유형
  - 사유
  - 상태
- 조치
  - limit
  - hide
  - archive
  - feature

## 9. 관리자 프로젝트 `/admin/projects`

### 기능

- 프로젝트 전체 목록 조회
- 상태 변경
- 검증 상태 확인
- 링크 상태 확인
- 중복 후보 확인

## 10. 관리자 피처드 `/admin/feature`

### 기능

- 홈 추천 프로젝트 편성
- 섹션별 순서 조정
- 피처드 시작/종료 일정 설정 optional

## 11. 관리자 작업 `/admin/jobs`

### 목적

- 운영 자동화 job을 수동으로 실행하고 결과를 즉시 확인한다.

### 기능

- `check-links`
- `cleanup-unclaimed`
- `recompute-ranking`
- 전체 job 순차 실행
- 최근 실행 결과 JSON 확인
- 권장 실행 주기 안내

## 12. 관리자 메일 `/admin/mail`

### 목적

- claim, 댓글 알림, 운영 상태 변경 메일의 최근 발송 기록을 운영자가 바로 확인한다.

### 기능

- 메일 템플릿 종류 표시
- 발송 상태 표시
  - queued
  - sent
  - failed
  - simulated
- 수신자, 제목, 생성 시각, 발송 시각 확인
- 메일 본문 텍스트 미리보기
- action URL 바로 열기

## 13. 공통 상태 명세

모든 주요 페이지는 아래 상태를 기본 지원해야 한다.

- 로딩 스켈레톤
- 빈 상태
- 404
- 서버 오류
- 제한/숨김 콘텐츠 안내
- 소유권 확인 대기 상태 안내
