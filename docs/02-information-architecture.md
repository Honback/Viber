## 2. 핵심 구조

이 서비스는 게시판이 아니라 아래 구조를 중심으로 동작한다.

- `Project`: 프로젝트 대표 엔티티
- `Project Activity`: 프로젝트에 귀속되는 활동
- `Comment`: 프로젝트 또는 활동에 달리는 피드백
- `Save`: 가벼운 북마크
- `Report`: 신고

즉, 탐색의 기본 단위도 프로젝트이고, 검색 결과도 프로젝트이며, 히스토리 역시 프로젝트 아래에 쌓인다.

## 4. 권한 정책

### 읽기

- 로그인 없이 허용한다.
- 검색과 태그 탐색도 무계정 허용이다.

### 프로젝트 등록

- 비회원도 가능하다.
- 단, 수정권을 위해 아래 중 하나는 필수다.
  - 이메일 매직링크
  - GitHub 인증

### 댓글

- visitor도 작성 가능하다.
- 단, `Turnstile` 또는 equivalent CAPTCHA와 더 강한 rate limit이 필요하다.
- guest alias와 guest fingerprint hash를 함께 저장해 완전 무제한 익명 상태로 두지 않는다.

### 저장

- 가벼운 계정이 필요하다.

### Feedback 활동

- 구조화된 `feedback` 활동은 인증된 member 이상이 작성할 수 있다.
- `Update`는 owner 또는 admin만 작성할 수 있다.
- owner 작성 `feedback`은 "Ask for Feedback" 성격이고, member 작성 `feedback`은 사용 후기/구조화 피드백 성격으로 구분한다.

### 공개 정체성

공개 화면에서는 아래만 노출한다.

- maker alias
- GitHub 연결 여부
- 도메인 검증 여부
- 선택적 외부 링크

실명은 강제하지 않지만, 내부 소유권은 확실하게 관리한다.

## 5. 주요 사용자 플로우

### 5-1. 방문자 탐색 플로우

1. 홈 또는 탐색 페이지 진입
2. 카드에서 프로젝트 가치와 상태 확인
3. `Try` 클릭 또는 상세 진입
4. 저장은 로그인, 댓글은 CAPTCHA 통과 후 바로 작성 가능

### 5-2. 비회원 등록 플로우

1. `/submit` 진입
2. 새 프로젝트 런치 작성
3. 기본 정보와 링크 입력
4. 소유권 검증 방식 선택
5. 발행
6. 소유권 확인 완료 후 즉시 게시
7. 추후 수정은 이메일 링크 또는 GitHub 소유권으로 처리

### 5-3. 프로젝트 소유자 업데이트 플로우

1. `/me/projects` 진입
2. 기존 프로젝트 선택
3. `Update` 또는 owner용 `Ask for Feedback` 추가
4. owner 권한 확인 후 즉시 반영
5. 프로젝트 상세의 Activity Feed에 누적

### 5-4. 멤버 피드백 플로우

1. 프로젝트 상세 진입
2. `Write Feedback` 선택
3. 로그인 후 구조화된 feedback 입력
4. member 권한과 anti-spam 검사
5. 정책에 따라 즉시 반영 또는 `pending`으로 적재

### 5-5. 신고 플로우

1. 프로젝트, 활동, 댓글에서 `Report` 실행
2. 사유 선택
3. 관리자 큐에 적재
4. 관리자 조치 기록 저장

- 메일 발송 기록 확인

## 7. URL 설계 원칙

- 프로젝트 상세는 사람이 읽기 쉬운 `slug` 기반 URL을 사용한다.
- 태그 역시 `slug` 기반으로 공개 URL을 갖는다.
- 등록 후 수정권 연결은 장기 노출 가능한 공개 URL이 아니라 `claim token` 기반으로 처리한다.
- 관리자 URL은 일반 사용자 UI와 명확히 분리한다.

## 8. 정보구조 상 주의점

- 프로젝트와 활동을 분리하지 않으면 업데이트 히스토리가 망가진다.
- 프로젝트 없는 "자유 글"이 들어오면 피드 품질이 급격히 떨어진다.
- 상세 페이지는 설명 문서가 아니라 체험 허브여야 한다.
- 탐색 페이지는 홈의 복사본이 아니라 검색과 필터 중심 페이지여야 한다.
