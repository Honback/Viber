# 콘텐츠 모델과 상호작용 정책

## 1. 콘텐츠 모델 개요

서비스 콘텐츠는 크게 다섯 종류로 구성된다.

- Project
- Project Activity
- Comment
- Save
- Report

이 중에서 최상위 공개 단위는 항상 `Project`다.

## 2. Project

프로젝트는 앱, 툴, 서비스의 대표 페이지다.

프로젝트는 아래 역할을 동시에 가진다.

- 쇼케이스 단위
- 탐색 결과 단위
- SEO 대상 페이지
- 활동 히스토리의 부모 단위

프로젝트 없는 독립 게시글은 허용하지 않는다.

## 3. Project Activity

프로젝트에 귀속되는 시간축 이벤트다.

### 허용 타입

- `Launch`
- `Update`
- `Feedback`

### 설계 의도

- 런치 이후 정보가 흩어지지 않게 한다.
- 사용자 피드백과 변화 기록을 같은 맥락 안에 남긴다.
- 단순 홍보 게시판처럼 동일 프로젝트가 여러 글로 분산되는 문제를 막는다.
- `feedback` 타입은 owner의 "Ask for Feedback"과 member의 구조화된 "Feedback"을 함께 수용한다.

## 4. 댓글 정책

### 허용 범위

- 프로젝트 전체 댓글
- 활동 단위 맥락을 가진 댓글
- 1단계 대댓글

### 규칙

- 무한 스레드는 허용하지 않는다.
- markdown-lite만 허용한다.
- 링크 수는 제한한다.
- 댓글 수정 가능 시간은 제한한다.
- 삭제는 soft delete로 처리한다.
- visitor 댓글은 CAPTCHA와 더 강한 rate limit을 전제로 한다.
- visitor 댓글은 `guest_name + guest_session_hash` 기반으로 저장한다.

### 하지 않는 것

- 완전 익명 댓글 전면 허용
- 실시간 채팅
- 다중 이모지 반응

## 5. 저장 정책

저장은 가벼운 북마크 기능으로 정의한다.

- 로그인 사용자만 가능하다.
- 프로젝트 단위로만 저장한다.
- 저장은 랭킹 신호로 사용한다.
- 저장 리스트는 개인 영역에서 다시 탐색 가능한 형태로 제공한다.

## 6. 신고 정책

신고는 프로젝트, 활동, 댓글 모두 대상으로 허용한다.

### 주요 신고 사유

- 스팸
- 중복
- 사칭
- 죽은 링크
- 부적절한 콘텐츠
- 허위 또는 오해 소지

### 기본 흐름

1. 신고 생성
2. 관리자 검토 대기
3. 조치 또는 기각
4. 조치 이력 저장

## 7. 소유권 모델

프로젝트 소유자는 아래 방식으로 연결된다.

- 이메일 매직링크
- GitHub 인증

비회원 등록을 허용하되, 소유권 없는 게시는 허용하지 않는다.

### 공개 표시

- maker alias
- GitHub 연결 여부
- 도메인 검증 여부

### 내부 관리

- edit token
- verification method
- primary owner 여부

## 8. 검증 상태

프로젝트에는 최소 아래 상태가 필요하다.

- `unverified`
- `github_verified`
- `domain_verified`

검증은 신뢰 신호로 사용하지만, 검증되지 않았다고 등록 자체를 막지는 않는다. 단, 운영 정책상 보류 또는 노출 제한 조건은 둘 수 있다.

## 9. 프로젝트 상태

프로젝트 공개 상태는 아래와 같이 관리한다.

- `pending`
- `published`
- `limited`
- `hidden`
- `rejected`
- `archived`

### 상태 의미

- `pending`: 소유권 확인 미완료 또는 자동 보호 hold 상태
- `published`: 일반 공개
- `limited`: 일부 노출 제한 또는 경고 포함 공개
- `hidden`: 공개 목록에서 숨김
- `rejected`: 자동 차단 또는 운영 거절
- `archived`: 오래되었거나 비활성 상태로 보관

## 10. 활동 상태

`project_posts` 역시 상태를 가져야 한다.

- `pending`
- `published`
- `hidden`

owner 작성 `Update`와 owner 작성 `Ask for Feedback`은 기본적으로 즉시 공개된다.
member 작성 `Feedback`은 운영 기준에 따라 즉시 공개 또는 `pending` 검토 대상으로 둘 수 있어야 한다.

## 11. 알림 정책

초기 알림은 이메일만 둔다.

### 발송 대상

- 프로젝트 소유자에게 새 댓글 알림
- 수정 링크 또는 매직링크
- 운영 제한 또는 상태 변경 안내

### 넣지 않는 것

- 앱 내 알림 센터
- 푸시 알림
- 팔로우 알림

## 12. 공유 정책

공유는 외부 확산을 위한 기본 기능으로 제공한다.

- 프로젝트 상세에서 공유 가능
- Open Graph 메타데이터 필수
- 카드/상세에서 대표 이미지와 한 줄 소개가 공유 품질을 좌우한다.

## 13. 콘텐츠 품질 최소 기준

프로젝트는 최소한 아래 조건을 만족해야 한다.

- 실제 접근 가능한 링크가 있어야 한다.
- 대표 이미지가 있어야 한다.
- 무엇을 하는지 이해 가능한 한 줄 소개가 있어야 한다.
- 설명이 너무 빈약하지 않아야 한다.
- 단순 광고성 랜딩 페이지만 존재하면 안 된다.

## 14. 제품 운영 관점에서 중요한 이유

이 콘텐츠 구조를 지키면 다음이 가능해진다.

- 프로젝트 중심 히스토리 축적
- SEO 가능한 상세 페이지 확보
- 스팸과 중복 관리 단순화
- 홈, 탐색, 랭킹, 저장 로직 일관성 유지

반대로 일반 게시판형 구조로 흐르면 프로젝트의 누적 가치가 사라진다.
